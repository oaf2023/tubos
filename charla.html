2. Dashboard sin datos — Base de datos recreada vacía

Usuario El dashboard no muestra datos. Agregué 3 ubicaciones, 3 cilindros y 2 clientes pero Dashboard dice 0 en todo.
Asistente Diagnóstico: prisma db push recreó la base de datos porque la ruta en .env apunta a prisma/db/custom.db (CLI), pero el runtime (start.js) usa db/custom.db (raíz). Al correr prisma db push, se creó una DB vacía en prisma/db/ mientras los datos reales estaban en db/. Solución: Copié db/custom.db a prisma/db/custom.db, luego corrí prisma db push que agregó columnas en lugar de recrear (ALTER TABLE). Ambos archivos sincronizados. Además, hice un commit de la DB (db/custom.db y prisma/db/custom.db) para preservar seed data en git.

4. Rutas no funcionan en la web — Distancias incorrectas

Usuario Las rutas no funcionan bien en la web, las distancias no son correctas.
Asistente Diagnóstico: al crear la ruta (POST /api/routes), se usaba Haversine (línea recta) en lugar de los valores OSRM calculados en la optimización. El frontend pasaba las paradas reordenadas pero NO las distancias reales. Fix:
  • El endpoint POST /api/routes ahora acepta distanciaReal y duracionReal pre-calculados
  • Si no se proveen, intenta OSRM automáticamente al crear la ruta
  • Timeout aumentado de 10s a 20s para OSRM público
  • Fallback a Haversine solo si OSRM falla
  • Frontend guarda optDistance después de optimizar y lo pasa al crear

5. Segunda ronda — Rutas siguen sin funcionar como Waze

Usuario No está funcionando bien, no crea rutas como Waze o Google Maps, no está optimizada.
Asistente Problemas raíz encontrados:
  1. TSP no incluía el origen (base) — la optimización empezaba desde points[0] (primera parada seleccionada), no desde la base. El TSP encontraba la ruta óptima entre paradas pero ignoraba la distancia desde la base.
  2. No se devolvía geometría OSRM — el mapa mostraba líneas rectas punteadas (Haversine) incluso cuando OSRM funcionaba.
  3. Geometría no se persistía — al recargar, las rutas guardadas se veían como líneas rectas.
  4. Frontend no enviaba el origen — optimizarRuta() solo enviaba points sin origen.
  5. Ruta planificada invisible — mientras se planificaba, no se mostraba ningún recorrido en el mapa hasta crear la ruta.
Fix completo: Endpoint /api/routes/optimize reescrito:
  • Ahora acepta origen: {lat, lng} y lo incluye en el TSP como punto fijo de inicio (índice 0)
  • Llama a osrmTable con todos los puntos (origen + paradas) para matriz real
  • Incluye origenPoint como último punto para la geometría de retorno
  • Resuelve TSP: [origen, parada₁, parada₂, ..., paradaₙ] → extrae solo paradas en orden
  • Devuelve geometría OSRM (geometry: [lat, lng][]) para dibujar en el mapa
  • Calcula distancia total incluyendo retorno a origen
Frontend optimizarRuta():
  • Ahora envía origen: { lat: base.lat, lng: base.lng }
  • Eliminado startIndex (innecesario con origen fijo)
  • Guarda routeGeometry de la respuesta
Prisma schema:
  • Nuevo campo geometry String? en modelo Ruta (JSON: [[lat, lng], ...])
POST /api/routes:
  • Acepta geometry del body y lo guarda como string JSON
Frontend crearRuta():
  • Pasa geometry al crear la ruta después de optimizar
  • Resetea routeGeometry y optDistance después de crear
Visualización en mapa:
  • rutaRoutes ahora incluye ruta en planificación (azul) cuando hay paradas seleccionadas
  • Si hay geometría OSRM → línea sólida; si no → punteada
  • rutaMarkers incluye marcadores de planificación + ruta guardada
  • Rutas guardadas parsean rutaSeleccionada.geometry desde JSON
Contribuciones (nuevo campo): Ruta.geometry, tipo Ruta actualizado con geometry?: string | null, MapMarker importado de map-view.
